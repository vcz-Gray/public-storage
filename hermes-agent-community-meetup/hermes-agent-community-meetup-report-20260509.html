전체 테마

1. 에이전트의 척추는 상태 머신단발 답변이 아니라 triage, ready, running, blocked, done이 남아야 사람이 이어받을 수 있습니다.
2. 메모리는 오염 방지가 어렵다저장량보다 stale update, source weighting, evidence boundary가 중요합니다.
3. 사람의 역할은 판단 지점으로 이동사람은 모든 지시를 반복하는 대신 merge, approval, blocked resolution, evidence gate를 판단합니다.
4. 웹은 협업 표면이 된다agent가 접근하기 쉬운 웹은 스크린 리더 사용자에게도 더 좋은 웹입니다.
5. 한국 맥락은 skill로 들어간다HWP, DART, 부동산, 법원 경매, 청약, 기상청 같은 한국형 도구가 실제 사용성을 만듭니다.
6. 실패는 기록되어야 한다Dispatch와 Kanban은 성공보다 실패 추적, escalation, 다음 plan 반영을 중요하게 다룹니다.

Opening & Hermes Agent 업데이트 설명

한국 첫 Hermes Agent Community Meetup의 문을 열며, Hermes를 “상태를 잃지 않는 에이전트 런타임”으로 설명한 세션.

핵심 포인트 보기
  • 홍콩 이후 한국에서 열린 첫 Hermes Agent Community Meetup으로, 신청자는 1,100~1,200명 수준이었다.
  • 행사는 슬라이드보다 데모 중심으로 설계되었고, 유튜브 라이브와 영어 번역/트랜스크립션이 함께 제공되었다.
  • 최근 업데이트의 축은 Kanban, Skill Curator, Nous Portal/모델 API, long-running goal 기능으로 정리된다.
  • Kanban은 triage/todo/ready/running/done/blocked 상태를 가진 에이전트 작업 상태 머신으로 소개되었다.
  • 긴 작업은 사람이 “계속해”라고 치는 방식이 아니라 judge model과 종료 조건으로 운영되어야 한다는 방향이 제시되었다.
에이전트의 척추는 상태 머신이다.

듣는 웹에서 협업하는 웹으로

시각장애 사용자의 스크린 리더 경험과 Hermes 브라우저 자동화를 연결해, agentic web 접근성을 새 기준으로 제안한 세션.

핵심 포인트 보기
  • 시각장애인은 화면 전체를 한 번에 보는 대신 스크린 리더로 웹을 순차적으로 “듣는다”.
  • 버튼/링크/초점/팝업/캡차/잘못된 태그는 단순 불편이 아니라 실제 업무 실패로 이어진다.

  • AI가 접근하기 쉬운 웹은 시각장애인도 접근하기 쉬운 웹이라는 확장된 접근성 관점이 제시되었다.
AI가 잘 접근할 수 있는 웹은 시각장애인들도 잘 접근할 수 있는 웹.

Ouroboros with Hermes Agent

Ouroboros를 통해 모호한 작업을 질문 루프와 계획 생성으로 분해하고, Hermes와 GitHub/Discord를 연결해 병렬 agentic workflow를 만드는 사례.

핵심 포인트 보기
  • Ouroboros는 현실 문제 모델링/시뮬레이션 benchmark에서 높은 성과를 냈다고 소개되었다.
  • 발표자는 Hermes 도입 후 iPad만 들고 다니며 agent가 PR/issue를 처리하게 하는 workflow를 공유했다.
  • Discord thread를 병렬 agent session처럼 활용해 여러 작업을 동시에 굴리는 방식이 강조되었다.
  • Ouroboros는 실행보다 “계획을 위한 계획”에 강한 harness로 설명되었다.
  • Socratic questioning으로 요구사항의 모호성을 줄이고 풍부한 issue/plan을 만드는 접근이 제시되었다.
Agentic workflow에서는 사람이 결정하기 전까지 다 해주도록 workflow를 짜야 한다.
Part 0215:25이재규

RLM-Forge, Hermes Agent

Hermes를 LLM 호출 경계로, Ouroboros를 재귀 실행 제어기로, Trace Guard를 evidence gate로 두는 실험적 runtime contract.

핵심 포인트 보기
  • RLM은 긴 입력을 작은 chunk/child call로 분해한 뒤 parent answer로 합성하는 재귀 언어 모델 개념이다.
  • RLM-Forge는 새 모델 학습이 아니라 Hermes runtime 위의 execution contract로 구현되었다.
  • Ouroboros는 decomposition/scheduling/trace replay/termination을 담당하고 Hermes는 실제 LLM call boundary가 된다.
  • Trace Guard는 fresh child manifest만 evidence로 인정하고 unsupported claim을 결정적으로 reject한다.
  • Hermes의 self-improving memory는 behavior prior로 유용하지만 evidence로 쓰이면 contamination을 만든다는 경고가 제시되었다.
The more self-improving a runtime is, the more you need a deterministic evidence gate.
Part 0215:40정승현

Introduce Hermes Kanban

Hermes Kanban을 durable work queue이자 multi-profile worker 운영판으로 소개한 세션.

핵심 포인트 보기
  • 기존 delegate_task는 작은 작업에는 유용하지만 긴 작업을 안정적으로 돌리기에는 한계가 있다.
  • Kanban은 SQLite 기반 task board이며 work queue로 쓰는 것이 가장 적합하다고 설명되었다.
  • triage는 곧바로 worker가 처리할 상태가 아니라, 대화를 통해 idea를 정리하는 공간이다.
  • Kanban Orchestrator는 큰 작업을 나누고, Kanban Worker는 ready task를 가져가 처리한다.
  • blocked 상태는 API key나 사람의 판단처럼 외부 입력이 필요한 작업을 보존하는 안전장치다.
  • 멀티 프로필은 session history와 skill 오염을 줄이는 실전 운영 팁으로 강조되었다.
Kanban을 work queue로 쓰는 게 제일 좋은 방안.
Part 0215:55조재표

Hermes에게 GitHub 관리시키기

GitHub webhook/event를 Hermes Kanban으로 받아 issue/PR triage, 개발, 리뷰, merge 판단으로 이어가려는 운영 실험.

핵심 포인트 보기
  • GitHub issue/PR event를 task queue에 넣고 Hermes가 triage부터 merge 판단까지 관리하는 구조가 제안되었다.
  • Cloudflare Tunnel은 로컬 서비스를 외부 도메인으로 노출하고 agent가 E2E test할 수 있게 하는 핵심 도구로 소개되었다.
  • AI에게 완전히 맡기기 전, 사람이 아는 제약을 명확히 알려줘야 한다. 예: GitHub는 webhook으로 받아야 한다.
  • Hermes에게 설계 누락/불명확점을 계속 질문하게 하는 manual questioning 패턴이 소개되었다.
  • AI native는 “귀찮다, 이걸 내가 왜 하지?”라고 느껴지는 일을 넘기는 사고방식으로 정의되었다.
  • 처음부터 완전자동화가 아니라 한 agent에게 잘 위임하는 단계부터 시작하라는 조언이 있었다.

Part 0316:25김효열

Agent는 만들었는데, 쓸모 있는지는 어떻게 알까?

Agent를 빨리 만드는 시대에, 실제로 사용될지 persona 기반 평가 루프로 미리 검증하는 방법을 제안한 세션.

핵심 포인트 보기
  • Prompt와 간단한 서버만으로 agent를 만드는 것은 쉬워졌지만, 사용자가 실제로 쓸지는 다른 문제다.
  • 배포 비용, 개인정보, 책임 소재, 정보 신뢰성은 agent service의 현실적 장애물이다.
  • Hermes evaluation loop에서 영감을 받아 Korean persona 기반 agent 평가 도구를 개발 중이라고 소개했다.
  • NVIDIA의 100만 persona dataset을 활용해 가상 소비자 반응을 평가하는 아이디어가 제시되었다.
  • YAML 형태의 agent/service 기획서를 persona별로 평가하고 markdown report로 되돌리는 흐름이 설명되었다.
  • Persona contamination을 줄이기 위해 persona별 독립 session/agent 평가가 필요하다고 답했다.
만드는 건 쉬운데, 정말 사용될 agent인지는 다른 문제.
Part 0316:40황신혁

Hermes x Membase: 더 잘 기억하는 에이전트 만들기

Hermes의 기본 memory 한계를 외부 universal memory layer로 보강하려는 Membase 플러그인 소개.

핵심 포인트 보기
  • Hermes 기본 memory는 memory.md와 user.md 중심이며 매우 작고 curated한 장기 컨텍스트다.
  • Markdown memory는 오래되거나 불필요한 정보가 prompt에 계속 들어가 context contamination을 일으킬 수 있다.
  • Membase는 vector DB + graph DB hybrid 방식으로 keyword/vector retrieval과 graph traversal을 결합한다고 소개되었다.
  • 선호 변화 같은 stale 정보는 outdated 처리하고 최신 정보를 우선해야 한다.
  • Slack, Gmail, Calendar, Notion 등 다양한 source에서 memory를 가져와 agent context로 활용할 수 있다.
  • 중요하지 않은 정보는 weight를 낮추고 중요한 결정/대화는 weight를 높이는 backend preprocessing 관점이 강조되었다.
중요한 정보가 저장되는 것보다 중요하지 않은 정보가 저장되어 contamination 되는 것을 막는 게 더 중요하다.

안드로이드 폰으로 Hermes agent on-live device 만들기

집에 굴러다니는 Android 폰을 Hermes용 always-listening voice bridge로 재활용하는 캐주얼한 실험.

핵심 포인트 보기
  • Solana Seeker phone gen 1처럼 업데이트가 끊긴 Android 기기를 버리지 말고 agent device로 쓰자는 발상에서 출발했다.
  • 목표는 Alexa처럼 wake word를 듣고 Hermes/backend에 전달해 답변을 말하는 장치다.
  • Android 앱은 LLM을 직접 돌리는 것이 아니라 mic/camera/speaker bridge 역할을 한다.
  • Backend brain은 local MLX, GPT, Mac mini, Tailscale 조합 등 원하는 환경으로 구성 가능하다.
  • Wake word → listen → recognize → answer → idle cycle이 제시되었다.
  • TTS latency와 데모 장비 reset/fallback UI가 현실적 병목으로 언급되었다.
이 친구의 역할은 브릿지예요.
Part 0317:10최재훈

Karpathy LLM Wiki: Hermes로 만드는 복리형 에이전트

Karpathy의 LLM Wiki 패턴을 Hermes built-in skill로 소개하며, RAG와 다른 “컴파일된 지식”의 운영 방식을 설명한 세션.

핵심 포인트 보기
  • RAG는 raw data를 chunking/search/reassemble하는 방식이라 매 query마다 지식이 instant하게 사라진다는 한계가 있다.
  • LLM Wiki는 논문, 기사, 문서, 코드 같은 raw source를 ingest해 entity, concept, comparison, query 페이지로 구조화한다.
  • 구조는 raw, entities, concepts, comparisons, queries, schema.md, log.md, index.md로 설명되었다.
  • Query 중 가치 있는 답은 다시 wiki에 저장되어 복리형 지식이 된다.
  • Obsidian graph view와 lint 기능은 orphan, broken link, 비대한 문서를 찾는 관리 도구로 제시되었다.
  • 완전 자동보다 사람이 schema와 방향을 잡고 Hermes가 구축하는 방식이 적합하다고 강조되었다.
입력을 digest해서 wiki로 만들어준다.

Kuma Studio Dispatch를 통해 16~18개 CLI agent를 동시에 돌리고, 성공보다 실패 추적과 ledger를 관리하는 시스템을 소개한 세션.

핵심 포인트 보기
  • Dispatch는 agent끼리 대화하고 작업 상태를 추적하는 orchestration/ledger 시스템이다.
  • 데모는 작은 게임/포트폴리오 리소스를 여러 agent가 동시에 만드는 형태였고, 발표자는 10분 내 성공보다 실패 추적을 보여주려 했다.
  • 메시지는 instruction, Q&A, status report, memo, blocked/failure, done 계열로 나뉜다.
  • Agent들은 tmux send 기반으로 Claude Code, Codex CLI 등 terminal agent를 혼합 제어할 수 있다.
  • 작업자 이름은 인격이 아니라 session tracking label로 설명되었다.
  • 실패는 상위자가 판정하기보다 worker가 스스로 fail을 보고하고 escalate하는 구조다.
  • 실패 기록은 ledger→plan update→self-learning→다음 dispatch로 이어진다.
에이전트를 관리할 에이전트가 필요하다.
